# Virelle Studios Security Policy

This policy covers the supported `main` branch and the production service deployed through Render.

## Reporting a vulnerability

Do not open a public issue for a suspected security vulnerability.

Email `leego972@gmail.com` with the subject:

```text
Virelle Security — <short summary>
```

Include affected endpoints or workflows, reproduction steps, expected and observed behaviour, and proof-of-concept material with all secrets and personal data redacted.

Virelle Studios aims to acknowledge reports within 72 hours and prioritise high-impact issues for prompt remediation. Public disclosure should wait until a reasonable remediation window has been agreed.

## Supported version

Only the current `main` branch and the active production deployment are supported. Historical commits and archived reports are not patched individually.

## Security baseline

### Secrets and credentials

- Production secrets are stored in Render environment variables.
- `.env` files, passwords, provider keys, private keys and live connection strings must never be committed.
- JWT and session signing use separate strong secrets.
- Client code receives only explicitly public values such as publishable Stripe keys.
- Provider credentials must not be logged or returned through API responses.
- A credential committed to Git must be revoked or rotated. Deleting the file does not remove the credential from history.

### Authentication and administrator authority

- Authentication state is established through the configured OAuth/session flow.
- Administrator authority is database-role based.
- Login, signup, environment variables and server startup must not automatically promote a user to administrator.
- Administrator role changes require a deliberate protected operation and audit logging.
- Lost administrator access must be recovered through an owner-approved database operation, not automatic bootstrap logic.

### Route protection and input validation

- Authenticated procedures validate the current user and resource ownership.
- Administrator procedures require an authenticated administrator.
- Destructive maintenance procedures remain disabled in production unless deliberately enabled for a bounded operation.
- External input is schema-validated before database or provider use.
- Database queries use parameter binding; untrusted values must not be interpolated into SQL identifiers or statements.
- Public authentication, upload and AI endpoints are rate limited.

### Browser and transport security

- Production traffic uses HTTPS.
- Session cookies use secure production attributes.
- Security headers are applied centrally, including content, framing and transport protections.
- Private media and compliance evidence must not be exposed through permanent public URLs.

## Payments and billing integrity

The Stripe webhook endpoint is:

```text
POST /api/stripe/webhook
```

Security requirements:

- The raw request body is preserved for Stripe signature verification.
- `STRIPE_WEBHOOK_SECRET` is required in production.
- Events are processed idempotently.
- Subscription renewals, credit purchases, wardrobe purchases and broadcast-minute purchases must not be fulfilled twice.
- Customer identity is resolved from trusted Stripe records; untrusted metadata is not sufficient authority.
- Failed or rejected work must release reservations or restore entitlements according to the relevant workflow.

Any change to webhook body parsing, event claiming, invoice/session fulfilment or customer resolution requires billing regression tests.

## AI providers and BYOK

- Provider keys are read from environment variables or the user's encrypted BYOK store.
- BYOK material remains server-side and is sent only to the selected provider.
- Logs and errors must redact credentials and sensitive provider responses.
- Ordinary direct broadcasting does not require an AI provider key.
- AI-assisted broadcasting requires the user's funded BYOK provider.
- Provider failure must not create an unrecoverable credit or minute charge.

## Adult Studio and compliance archive

Adult Studio requires stricter controls than the standard workspace:

- age and identity verification where required;
- consent and depicted-person safeguards;
- prohibition of minors, non-consensual content and prohibited public-figure use;
- managed broadcasting with recording;
- private compliance retention;
- short-lived signed access for authorised review;
- retention and legal-hold rules that cannot be bypassed through the client.

The compliance bucket must be private and encrypted with no public-read policy.

## Storage and uploads

- Uploads are size- and type-validated.
- Object keys are generated by the server rather than trusted from user paths.
- Signed URLs use bounded expiry.
- Application media and compliance evidence use appropriate, separate access policies.
- Failed uploads or provider jobs must not leave completed records pointing to unavailable objects.

## Logging and incident evidence

Security-relevant events include:

- administrator actions and role changes;
- blocked maintenance attempts;
- authentication abuse and rate-limit events;
- Stripe webhook and fulfilment failures;
- provider authentication failures;
- broadcast reservation, acceptance and release events;
- compliance archive failures;
- suspected credential exposure.

Logs must identify the operation and affected resource without reproducing secrets, full tokens, private media URLs or unnecessary personal data.

## Dependency and build security

Pull requests and `main` are protected by automated checks including:

- TypeScript validation;
- unit and integration tests;
- production build;
- dependency audit at the repository's configured threshold;
- secret-pattern scanning;
- Render container validation;
- desktop and mobile parity checks where applicable.

Dependency overrides and advisory allowlists must be narrowly scoped, documented in configuration and revisited when upstream fixes become available.

Local verification:

```bash
pnpm verify
```

## Credential exposure response

When a credential is exposed:

1. Revoke or rotate it at the provider immediately.
2. Update the corresponding Render environment variable.
3. Redeploy and verify the affected integration.
4. Search the current repository for duplicate copies.
5. Review provider and application access logs where available.
6. Record the incident without reproducing the secret.
7. Consider history rewriting only as additional containment, never as a substitute for rotation.

## Security-sensitive changes

The following changes require focused review and regression coverage:

- authentication, sessions, OAuth or administrator roles;
- Stripe webhooks, subscriptions, credits or fulfilment;
- BYOK encryption, credential resolution or provider dispatch;
- uploads, signed URLs or object permissions;
- Adult Studio verification, consent or policy enforcement;
- broadcast recording, minute reservation or compliance callbacks;
- database migrations affecting security or billing records;
- security headers, rate limits or audit logging.
